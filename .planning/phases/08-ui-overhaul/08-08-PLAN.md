<pitfalls>

- **Do NOT delete `TrackingStatusLayout` or `TrackingErrorLayout`** â€” the rewritten HeroRecordCard reuses both. Only `TrackingIdleLayout` is removed.
- **Do NOT make `_handleStart` await `notifier.start()`** â€” per `.planning/debug/hero-start-double-tap.md` Resolution: awaiting blocks the dashboard for ~500ms-1s during foreground-service spin-up. Fire-and-forget. The notifier transitions to TrackingStarting synchronously, which the hero immediately reflects.
- **Notifier re-entry is safe** â€” `TrackingNotifier.start()` (lines 292-302 of tracking_providers.dart) guards against being called from any state other than Idle/Error. Double-tap on the hero is harmless.
- **Permission flow stays in DashboardScreen** â€” do NOT move permission preflight into HeroRecordCard. Separation of concerns: hero displays state, dashboard owns permission UX.
- **InProgressCard becomes redundant but should remain** â€” when the user scrolls past the hero, the in-progress card under TodaySection still shows the elapsed time. Make it inert (`onTap: null`) rather than removing the widget â€” removal would change layout/scroll behaviour and isn't required for the gap.
- **`flutter_background_service` Stop intent** â€” the notification action uses `showsUserInterface: true` which routes through `Activity.onNewIntent` and brings the app to the foreground.
