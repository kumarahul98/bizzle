Output: a v5 schema + flag column + setter, a no-flash root gate, a LoginScreen with a visible Skip, a verified sync-backlog reconciliation in `signIn()`, and an automated migration + gate + backlog test suite. Plan 02 (wave 1) adds the guest "not connected" indicator on top of this.

## Critical investigation findings (pinned from the real code — DO NOT re-derive)

1. **Are guest trips enqueued at save time or only on sign-in?** AT SAVE TIME, unconditionally. `tracking_service_controller.dart` (~line 234) calls `_syncQueueDao.enqueueCreate(trip.id)` inside the persist transaction with NO auth gate; the manual-entry / edit / delete paths (`trip_management_providers.dart` lines ~135/161/215) likewise enqueue unconditionally. The sync ENGINE gates on `AuthSignedIn` (`sync_engine.dart` step (c), line ~117) so guest items simply never flush until sign-in. ⇒ D-09 "enqueue on first sign-in" is NOT needed — the backlog already exists in `sync_queue` as pending rows.

2. **Is userId embedded in the sync_queue payload?** For create/update: NO — `enqueueCreate`/`enqueueUpdate` store a NULL payload; the engine re-reads the fresh trip row at flush (`_tripsDao.findById`) and serializes via `TripSerializer.toJson`, which **OMITS `userId` entirely** (the server forces uid from the verified token). For delete: the payload is `{id,userId}` JSON built in `trip_management_providers.dart` (~line 156) and **hardcodes `kDefaultUserId`** — BUT the engine's delete path (`_api.deleteTrip(entry.key)`) uses ONLY the tripId in the URL; the payload `userId` is never put on the wire, and the server tombstones by uid-from-token.

**Net effect on D-08:** the guest backlog ALREADY syncs correctly with the real uid after sign-in (engine un-gates → trip rows are backfilled → re-read picks them up → userId isn't even sent for upserts and is ignored for deletes). The remaining work is to (a) make the client pristine — rewrite the stale `local_user` inside any pending delete payload to the real uid during the sign-in transaction so no `local_user` reference survives anywhere — and (b) PIN the exactly-once invariant with a test (the safety net the context mandates). Approach per Claude's Discretion: **re-serialize the pending delete payload's userId in place** (cheap, targeted), NOT delete+re-enqueue, because create/update payloads are already userId-free and need no touch.
</objective>

sync_queue / userId reconciliation: the SyncQueueDao (lib/database/daos/sync_queue_dao.dart) owns the queue. The delete payload is `{"id":..,"userId":"local_user"}` JSON. Add a reconciliation method:
```dart
/// SC#4 / D-08: rewrite the embedded userId in every PENDING delete payload
/// from [kDefaultUserId] to [newUserId] so a guest's queued deletes carry
/// the real uid after sign-in. create/update rows store a NULL payload (the
/// engine re-reads the freshly-backfilled trip row at flush, and the
/// serializer omits userId), so they need no rewrite. Idempotent: a payload
/// already at the real uid is left untouched. Returns the count rewritten.
Future<int> reconcilePendingUserId(String newUserId) async {
  final rows = await (select(syncQueue)
        ..where((q) => q.status.equals(kSyncStatusPending))
        ..where((q) => q.action.equals(kSyncActionDelete)))
      .get();
  var changed = 0;
  for (final row in rows) {
    final raw = row.payload;
    if (raw == null) continue;
    final map = jsonDecode(raw) as Map<String, dynamic>;
    if (map['userId'] == kDefaultUserId) {
      map['userId'] = newUserId;
      await (update(syncQueue)..where((q) => q.id.equals(row.id)))
          .write(SyncQueueCompanion(payload: Value<String>(jsonEncode(map))));
      changed++;
    }
  }
  return changed;
}
```
