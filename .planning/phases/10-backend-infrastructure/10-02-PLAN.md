<objective>
Implement the three HTTPS REST handlers for the Commute Tracker backend and mount
them on the Express app created in Plan 01:

- `POST /trips/sync` — batch-upsert trips (BACK-02)
- `DELETE /trips/:tripId` — soft-delete a trip (BACK-03)
- `GET /trips/restore` — return the caller's non-deleted trips (BACK-04)

Each handler follows the **verify → validate → trust** contract: verify the
Firebase ID token first (401 on missing/invalid/expired, no further work), then
zod-validate, then operate on trusted typed data. Server forces ownership: the
authenticated `uid` is authoritative (D-08). Responses use the consistent shape
`{ statusCode, body: { data?, error? } }` (D-06).

Purpose: This is the functional core of the backend — the three endpoints the
Phase 11 sync engine will call. Plan 01 stood up the scaffold + shared
utils/types + Express skeleton; this plan makes the routes actually do work.

Output: three self-contained handler files + an updated `index.ts` router, all
compiling clean under strict TS.
</objective>

  <behavior>
    - No Authorization header → 401 {error}, zero Firestore writes.
    - Invalid/expired token → 401 {error}, zero Firestore writes.
    - Valid token + invalid body (missing trips, trips not array, malformed trip) → 400 {error}.
    - Valid token + valid {trips:[...]} → each doc written to trips/{trip.id} with
      userId forced to the token uid, deleted:false, serverUpdatedAt=serverTimestamp();
      response 200 {data:{syncedIds:[...trip ids...]}}.
    - Client-supplied userId on any trip is ignored/overwritten with the token uid (D-08).
    - >500 trips → split into multiple batches of ≤500, all committed (D-12).
    - Re-sending the same trips is idempotent (set merge:true keyed by UUID).
  </behavior>
  <action>
Create `backend/functions/src/handlers/sync-trips.ts`. Export an async Express
handler `syncTripsHandler(req, res)`.

Order is mandatory (verify → validate → trust, per D-07):
1. **Auth first.** `const uid = await verifyRequestAuth(req)`. Wrap in try/catch;
   on AuthError (or auth failure) respond `401 { error: "Unauthorized" }` and
   `return` — do NO Firestore work. Do not leak the token or exception detail.
2. **Validate.** `const parsed = syncTripsBodySchema.safeParse(req.body)`. If
   `!parsed.success` respond `400 { error: "Invalid request body" }` and return.
3. **Trust + write.** From here data is typed. For each `trip` in
   `parsed.data.trips`, build the persisted doc:
   - **FORCE `userId: uid`** (ignore/overwrite any client-supplied userId — D-08).
   - `deleted: false`, `deletedAt: null`, `serverUpdatedAt: FieldValue.serverTimestamp()` (D-10/D-11).
   - keep all trip fields (ISO-string timestamps stored as received — D-10).
   Write via chunked Firestore **batched writes** (D-12): chunk `trips` into
   groups of ≤500 (Firestore batch op limit). For each chunk, create a batch via
   `getFirestore().batch()`, then for each trip `batch.set(tripsCollection().doc(trip.id), doc, { merge: true })`,
   then `await batch.commit()`. Commit chunks sequentially (or `Promise.all` over
   chunk commits — sequential is fine and simplest).
   Define the chunk size as a named const (e.g. `const FIRESTORE_BATCH_LIMIT = 500`)
   — no magic numbers.
4. Respond `200 { statusCode: 200, body: { data: { syncedIds: parsed.data.trips.map(t => t.id) } } }`.

Wrap the write phase in try/catch; on unexpected failure respond
`500 { error: "Internal error" }` (no stack trace). Strict TS, no `any` — type
the built document as `TripDocument` (or `WithFieldValue<TripDocument>` where the
serverTimestamp sentinel is used). Import `FieldValue`/`getFirestore` from
`firebase-admin/firestore`. Use `tripsCollection()` and `tripConverter` from
`utils/firestore.ts` so writes go through the typed converter.

This satisfies BACK-02 success criteria 1 (accepts batch + writes) and 4 (rejects no/invalid token).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → `api` HTTPS function | Untrusted: token, request body, and path params all cross here. Anyone on the internet can hit the endpoint. |
| `api` function → Firestore | Trusted: Admin SDK; Security Rules deny-all to clients (D-13, Plan 01), so only verified-and-validated server code touches data. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Spoofing | all 3 handlers (request identity) | mitigate | `verifyRequestAuth` runs as the FIRST line; missing/invalid/expired token → 401 before any Firestore access (D-07). |
| T-10-02 | Tampering | sync-trips.ts (client-supplied userId) | mitigate | Server FORCES `userId = uid` from the verified token; client userId ignored/overwritten (D-08). Prevents writing trips into another user's namespace. |
| T-10-03 | Tampering / Elevation | delete-trip.ts (deleting others' trips) | mitigate | Read doc, compare `data.userId !== uid` → 404; only owner can soft-delete (D-08). |
| T-10-04 | Information Disclosure | restore-trips.ts (reading others' trips) | mitigate | Query filtered by `where("userId","==",uid)`; cannot return another user's data (D-08). |
| T-10-05 | Information Disclosure | all handlers (error responses) | mitigate | Responses return short typed messages only; NEVER stack traces or token values (D-06). |
| T-10-06 | Information Disclosure | delete-trip.ts (existence oracle) | mitigate | Foreign-owned trip returns 404 (same as missing), not 403 — does not reveal that another user's trip exists. |
| T-10-07 | Denial of Service | sync-trips.ts (huge trips array) | accept | v0.1: single-user client sync batches are small; chunking at ≤500/batch (D-12) bounds per-commit cost. No explicit array-length cap for v0.1; revisit if abused. Low-value personal-data target. |
| T-10-08 | Repudiation | all writes | accept | `serverUpdatedAt = serverTimestamp()` stamped on every write (D-10) gives a basic audit trail; full audit logging out of scope for v0.1. |
| T-10-09 | Tampering | input shape (non-schema fields) | mitigate | zod validation at handler entry rejects malformed bodies/params with 400 before any trust (D-07). |
</threat_model>
