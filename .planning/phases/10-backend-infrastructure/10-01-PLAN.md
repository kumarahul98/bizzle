<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client to api (HTTPS Cloud Function) | Untrusted client request (Authorization header + JSON body) crosses into the backend. This plan builds the verification/validation gates the boundary depends on; handlers wire them in Plan 02. |
| api to Firestore | Admin SDK writes; clients are denied by deny-all rules (this plan). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Spoofing | Authorization header / verifyAuth (auth.ts) | mitigate | `verifyAuth` requires a `Bearer` token and calls `getAuth().verifyIdToken`; missing/invalid/expired throws `AuthError` (401). Foundation for all handlers (D-07). |
| T-10-02 | Tampering | Request body (validation.ts) | mitigate | zod `tripSchema`/`syncTripsBody`/`tripIdParam` validate every field at handler entry before any write (D-07). |
| T-10-03 | Information disclosure | Direct Firestore access from client | mitigate | `firestore.rules` deny-all (`allow read, write: if false`); only the Admin SDK reads/writes (D-13, success criterion 5). |
| T-10-04 | Information disclosure | Error responses (response.ts / AuthError) | mitigate | `sendError` returns only a string message; `AuthError` never echoes the token or stack trace (D-06). |
| T-10-05 | Elevation of privilege | Client-supplied userId | mitigate | `tripSchema.userId` is optional and ignored; the server forces the token uid on write (D-08). Schema established here; enforcement lands in Plan 02 handlers. |
| T-10-06 | Repudiation | Audit of writes | accept | `serverUpdatedAt` (FieldValue.serverTimestamp) is added to `TripDoc` for write audit; full audit logging is out of scope for v0.1. |
</threat_model>
