<harness_patterns>
Use these EXACT patterns (the supertest-against-exported-app + emulator-REST mint approach,
RESEARCH "Emulator Testing" Approach A).

1. mintIdToken(uid) — obtain an emulator ID token verifyIdToken() accepts for a KNOWN uid.
   The Auth emulator exposes the Identity Toolkit REST surface. createCustomToken → exchange
   via `accounts:signInWithCustomToken` returns an UNSIGNED idToken whose `uid` equals the
   chosen uid; the Admin SDK accepts it when FIREBASE_AUTH_EMULATOR_HOST is set.

   ```typescript
   // backend/functions/test/helpers/mint-token.ts
   import { getAuth } from "firebase-admin/auth";
   const AUTH_HOST = process.env.FIREBASE_AUTH_EMULATOR_HOST!; // e.g. "127.0.0.1:9099"

   async function ensureUser(uid: string): Promise<void> {
     try { await getAuth().createUser({ uid }); }
     catch (e) { /* swallow auth/uid-already-exists */ }
   }

   export async function mintIdToken(uid: string): Promise<string> {
     await ensureUser(uid);                                  // user at the exact uid (Admin SDK → emulator)
     const customToken = await getAuth().createCustomToken(uid);
     const res = await fetch(
       `http://${AUTH_HOST}/identitytoolkit.googleapis.com/v1/accounts:signInWithCustomToken?key=fake-api-key`,
       {
         method: "POST",
         headers: { "Content-Type": "application/json" },
         body: JSON.stringify({ token: customToken, returnSecureToken: true }),
       },
     );
     const json = (await res.json()) as { idToken?: string; error?: unknown };
     if (!json.idToken) throw new Error(`mintIdToken failed: ${JSON.stringify(json)}`);
     return json.idToken; // verifyIdToken accepts this; reported uid === the chosen uid
   }
   ```
   Rationale for signInWithCustomToken (not anonymous signUp): we need a KNOWN uid per
   test (userA / userB) so ownership assertions are deterministic. createCustomToken +
   exchange yields an ID token whose `uid` equals the chosen uid. Admin SDK custom-token
   signing works against the emulator with no real service-account credentials when
   FIREBASE_AUTH_EMULATOR_HOST is set.

2. Admin SDK + emulator wiring (emulator.ts):
   ```typescript
   process.env.FIRESTORE_EMULATOR_HOST ??= "127.0.0.1:8080";
   process.env.FIREBASE_AUTH_EMULATOR_HOST ??= "127.0.0.1:9099";
   process.env.GCLOUD_PROJECT ??= "travey-298a7";
   import { initializeApp, getApps } from "firebase-admin/app";
   import { getFirestore } from "firebase-admin/firestore";
   if (!getApps().length) initializeApp({ projectId: process.env.GCLOUD_PROJECT });
   export const db = getFirestore();
   export async function clearFirestore(): Promise<void> {
     // delete all docs in trips collection between tests
     const snap = await db.collection("trips").get();
     await Promise.all(snap.docs.map((d) => d.ref.delete()));
   }
   export async function seedTrip(partial): Promise<void> { /* db.collection("trips").doc(id).set(...) */ }
   ```
   NOTE: env vars must be set BEFORE firebase-admin is imported/initialized — set them at
   the very top of emulator.ts (or a jest setupFiles entry) so the Admin SDK picks up the
   emulator endpoints on init.

3. supertest against the EXPORTED Express app (in-process — no HTTP server, no functions

   import { app } from "../../src/index"; // the bare Express app exported by Plan 02
