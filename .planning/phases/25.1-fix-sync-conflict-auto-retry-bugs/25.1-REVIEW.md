## Warnings

### WR-01: Conflict resolution writes trips without enqueueing a sync update

**File:** `lib/features/settings/widgets/conflict_resolution_sheet.dart:45,71`
**Issue:** `_applyAll` calls `tripsDao.updateTrip(...)` for both `Use cloud` and `Merge` actions but never calls `SyncQueueDao.enqueueUpdate`. Every other `updateTrip` call site in the codebase pairs the write with an enqueue (`backfill_provider.dart:43+50`, `trip_management_providers.dart:102+149`), and `TripsDao.deleteTrip`'s doc comment mandates the same pairing for deletes. Consequence: a **merged** trip is a new field combination that exists nowhere in Firestore — it is never pushed, so the cloud backup silently diverges and a future restore would lose the merge. (For `Use cloud` the payload matches the cloud copy, so the impact is limited to the stale `updatedAt`.)
**Fix:**
```dart
if (action == kConflictUseCloud) {
  await tripsDao.updateTrip(companion);
  await ref.read(syncQueueDaoProvider).enqueueUpdate(conflict.localTrip.id);
  resolvedCount++;
} else if (action == kConflictMerge) {
  // ... build merged ...
  await tripsDao.updateTrip(merged);
  await ref.read(syncQueueDaoProvider).enqueueUpdate(localTrip.id);
  resolvedCount++;
}
```
Ideally wrap each `updateTrip` + `enqueueUpdate` pair in `db.transaction()` to match the atomicity convention used by the edit flow. (Pre-existing — not introduced by this phase, but in a reviewed file.)

### WR-02: `ref` used after `await` without a lifecycle guard in `_applyAll`

**File:** `lib/features/settings/widgets/conflict_resolution_sheet.dart:76-78`
**Issue:** `_applyAll` awaits multiple DAO writes in a loop, then calls `ref.read(restoreControllerProvider.notifier).resolveConflicts(...)`. The `mounted` check at line 79 protects only the `Navigator.pop`. A modal bottom sheet can be drag-dismissed while the awaits are in flight; using `ref` after the `ConsumerState` is disposed throws (`Cannot use "ref" after the widget was disposed`), which would surface as an unhandled async error and skip `resolveConflicts`, leaving the restore controller stuck in its conflict state.
**Fix:**
```dart
if (!mounted) return;
ref.read(restoreControllerProvider.notifier).resolveConflicts(resolvedCount);
Navigator.of(context).pop();
```
(Also consider `isDismissible: false` on the sheet while applying, since a mid-loop dismissal leaves the batch partially applied with no controller notification.)

### WR-03: Merge only lets the user pick 5 fields — the rest silently take cloud values

**File:** `lib/features/settings/widgets/conflict_resolution_sheet.dart:52-70`
**Issue:** `merged` is built via `cloudTrip.copyWith(...)`, so every field NOT in the 5-field selection list (`timeMovingSeconds`, `timeStuckSeconds`, `routePolyline`, `isManualEntry`, `createdAt`, `userId`, ...) comes from the cloud copy regardless of selections. With the new all-`local` default, "Merge All" with nothing tapped produces a row where `durationSeconds`/`startTime`/`endTime` are local but `timeMovingSeconds`/`timeStuckSeconds` are cloud — the traffic split can exceed or mismatch the trip duration, corrupting the stats screens that aggregate those columns. The D-05 flip makes this hybrid the default outcome rather than an opt-in.
**Fix:** Derive non-selectable stats fields from whichever side won the related selectable field, at minimum tying `timeMovingSeconds`/`timeStuckSeconds` to the `durationSeconds` selection:
```dart
final durationFromLocal =
    (selections['durationSeconds'] ?? 'local') == 'local';
// ...
timeMovingSeconds: durationFromLocal
    ? drift.Value(localTrip.timeMovingSeconds)
    : cloudTrip.timeMovingSeconds,
timeStuckSeconds: durationFromLocal
    ? drift.Value(localTrip.timeStuckSeconds)
    : cloudTrip.timeStuckSeconds,
```

### WR-04: D-07 tests race the post-save trigger against `markFailed` (flaky pattern)

**File:** `test/unit/sync/sync_engine_test.dart:661-664,684-686`
**Issue:** In "second trigger within window does not re-fire retryFailed" and "trigger after window elapses DOES re-fire retryFailed", the engine's `start()` has already attached the `watchPending()` listener when the test runs `await queueDao.enqueueCreate('t2'); await queueDao.markFailed(t2Id);`. The enqueue fires the rising-edge trigger (`unawaited(processPending())`), which races the subsequent `markFailed`. If the drain's `getPending()` wins the race, `t2` is synced *before* being marked failed — `api.syncCalls` grows by one and the `expect(api.syncCalls.length, callsAfterFirst)` assertion in the first test fails intermittently. It currently passes because the drain's async gaps (`await _isOnline()`) usually lose the race, but the ordering is not guaranteed.
**Fix:** Neutralize the post-save trigger for these two tests — either enqueue+fail inside a single transaction so the stream emission observes the terminal state:
```dart
final t2Id = await db.transaction(() async {
  final id = await queueDao.enqueueCreate('t2');
  await queueDao.markFailed(id);
  return id;
});
```
or add `await _pumpEventQueue();` between enqueue and the assertion baseline and recapture `callsAfterFirst` after `markFailed`.

### IN-06: Stuck banner appears on the very first failure and never updates reactively

**File:** `lib/features/dashboard/screens/dashboard_screen.dart:193-194`, `lib/sync/sync_engine.dart:105-107`
**Issue:** Two semantics are conflated in `autoRetryWindowElapsed`: for the retry gate, `_lastAutoRetry == null` correctly means "gate open, retry now"; but for the banner it means "show 'sync stuck' immediately on a fresh `SyncFailed` even though no auto-retry has been attempted yet" — the banner reads as "stuck >4h" when it may be seconds old (e.g. a first transient 503, which also emits `SyncFailed` with `countFailed()==0`). Additionally, `_StuckBannerGate` reads the getter once via `ref.read` at build time, so the banner does not appear/disappear when the window elapses without an unrelated rebuild. Not a regression from this phase (the rename preserved behavior), but the D-04 rename discussion is the right context to decide whether the banner should require `_lastAutoRetry != null`.
**Fix:** If the banner is meant to signal a *persistent* stall, gate on "an auto-retry has happened AND the window has since elapsed AND failed rows remain" via a dedicated getter (e.g. `bool get stuckBeyondAutoRetryWindow => _lastAutoRetry != null && _now().difference(_lastAutoRetry!) > kFailedAutoRetryWindow;`). Otherwise document the intentional first-failure display at the call site.

### IN-07: Catch-all in `processPending` can itself throw

**File:** `lib/sync/sync_engine.dart:147`
**Issue:** The `on Object` handler runs `_status.set(SyncFailed(await _failedCount()))`; if `_failedCount()` throws (e.g. DB closed during teardown while a fire-and-forget drain is in flight), the exception escapes the catch block into the `unawaited(...)` caller as an unhandled async error — defeating the "never rethrows into the UI" contract (T-11-02-01).
**Fix:**
```dart
} on Object {
  var failed = 0;
  try {
    failed = await _failedCount();
  } on Object {
    // DB unavailable — surface a generic failure with count 0.
  }
  _status.set(SyncFailed(failed));
}
```
