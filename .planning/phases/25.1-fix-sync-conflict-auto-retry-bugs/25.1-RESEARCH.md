### Pattern 1: Testing the auto-retry gate via the injected clock + `handleResume()`
**What:** `SyncEngine`'s constructor takes `DateTime Function() now` (default `DateTime.now`).
Tests already override this with a mutable `late DateTime clock;` closure variable
(`sync_engine_test.dart:573,595,603`). `handleResume()` is `@visibleForTesting` and runs the
EXACT gate logic the connectivity-restore closure runs (byte-identical condition today; will
call the shared predicate after D-04).
**When to use:** Any assertion about the 4h gate (D-07's three assertions).
**Example:**
```dart
// Source: existing test/unit/sync/sync_engine_test.dart pattern, extended per D-07
test('second trigger within window does not re-fire retryFailed', () async {
  await tripsDao.insertTrip(_trip('t1'));
  final t1Id = await queueDao.enqueueCreate('t1');
  await queueDao.markFailed(t1Id); // simulate an exhausted-retry row

  await engine.start();
  await engine.handleResume(); // _lastAutoRetry null -> retryFailed() fires,
                                 // resetFailed() -> t1 pending -> drains -> synced
  await _pumpEventQueue();
  expect(await rowStatus(t1Id), kSyncStatusSynced);
  final callsAfterFirst = api.syncCalls.length;

  // A second failed row appears AFTER the auto-retry stamp, within the window.
  await tripsDao.insertTrip(_trip('t2'));
  final t2Id = await queueDao.enqueueCreate('t2');
  await queueDao.markFailed(t2Id);

  await engine.handleResume(); // clock unchanged -> still inside kFailedAutoRetryWindow
  await _pumpEventQueue();

  // No resetFailed happened for t2 -> it is still 'failed', not drained.
  expect(await rowStatus(t2Id), kSyncStatusFailed);
  expect(api.syncCalls.length, callsAfterFirst); // zero new network calls
});
```

### Pattern 2: Advancing the fake clock past the window
**What:** Because `clock` is a mutable variable captured by the `now: () => clock` closure,
reassigning it between actions changes what `_now()` returns on the next call — no `Timer`
or `fakeAsync` package needed.
**Example:**
```dart
// Continuing from Pattern 1's t2 (still 'failed', clock unchanged)
clock = clock.add(kFailedAutoRetryWindow + const Duration(minutes: 1));

await engine.handleResume(); // difference > kFailedAutoRetryWindow -> retryFailed() fires
await _pumpEventQueue();

expect(await rowStatus(t2Id), kSyncStatusSynced);
```

### Pattern 3: Manual retry always stamps the window (pins D-03)
**Example:**
```dart
test('manual retryFailed stamps the window, suppressing a following auto trigger', () async {
  await tripsDao.insertTrip(_trip('t1'));
  final id = await queueDao.enqueueCreate('t1');
  await queueDao.markFailed(id);

  await engine.retryFailed(); // manual "tap to retry" — always runs, stamps _lastAutoRetry
  await _pumpEventQueue();
  expect(await rowStatus(id), kSyncStatusSynced);

  await tripsDao.insertTrip(_trip('t2'));
  final t2Id = await queueDao.enqueueCreate('t2');
  await queueDao.markFailed(t2Id);

  await engine.handleResume(); // same clock, inside window -> suppressed
  await _pumpEventQueue();
  expect(await rowStatus(t2Id), kSyncStatusFailed);
});
```

### Anti-Patterns to Avoid
- **Simulating the real `EventChannel` for the auto-retry gate test:** Adds
  `MockStreamHandler` setup complexity and a `.distinct()` footgun (a second identical
  `['wifi']` event is silently swallowed by `connectivity_plus`'s wrapper) for zero additional
  coverage over calling `handleResume()` directly, since both trigger closures share the same
  gate condition (and will share the literal same predicate call after D-04).
- **Re-testing `_applyAll`'s non-merge branches:** `kConflictKeepLocal` / `kConflictUseCloud`
  paths are already covered by the first two existing widget tests and are untouched by this
  phase — do not duplicate that coverage.
- **Asserting on `_mergeSelections` internal state:** It is a private `State` field; assert on
  the resulting Drift row (`db.tripsDao.findById(...)`) as the existing tests already do, not
  on widget internals.

## Common Pitfalls

### Pitfall 1: Believing the roadmap's stale premise and reimplementing fixed code
**What goes wrong:** Following the original SC1/SC2 wording literally would mean rewriting
`retryFailed()`'s stamp logic and rebuilding a merge UI that already exists, silently
reverting/duplicating working code (or, worse, introducing a regression into code that Phase 24
already shipped correctly in the v0.3 merge).
**Why it happens:** Phase 24's `24-VERIFICATION.md` (2026-06-16) is dated before the v0.3 merge
to main (2026-07-06, commit `54f70eb`) that fixed both defects — the roadmap text was never
updated after the merge.
**How to avoid:** Trust `25.1-CONTEXT.md`'s reframing (D-01) over `ROADMAP.md`'s literal SC1/SC2
wording; this RESEARCH.md independently re-confirmed both fixes are present via direct file read
and `git show 54f70eb --stat`.
**Warning signs:** A plan task titled "implement field-by-field merge" or "add `_lastAutoRetry`
assignment" — both already exist; any such task is out of scope.

### Pitfall 2: `.distinct()` swallowing a "second trigger" test on the real connectivity stream
**What goes wrong:** `connectivity_plus`'s top-level `Connectivity().onConnectivityChanged`
(`connectivity_plus.dart:51`) applies `.distinct((a, b) => a.equals(b))` before your listener
ever sees it. A test that pushes `['wifi']` twice via `MockStreamHandler` to prove "a second
trigger within the window is suppressed" will see the SECOND event never delivered at all — the
test would pass for the wrong reason (no event delivered, not "gate suppressed it").
**Why it happens:** The `.distinct()` operator lives in the public `connectivity_plus` package
wrapper, one layer above the `EventChannel`/platform-interface layer that `MockStreamHandler`
mocks.
**How to avoid:** Prefer `handleResume()` direct calls (Pattern 1-3 above) for the gate-contract
assertions, since they bypass the stream entirely. If connectivity-specific coverage is still
desired, alternate the mocked connectivity value (`['none']` then `['wifi']` then `['none']`
then `['wifi']` again) so consecutive delivered events are non-identical and `.distinct()`
doesn't interfere — added complexity for marginal benefit, per the "Alternatives Considered"
table above.
**Warning signs:** A connectivity-trigger test that passes even when you comment out the gate
check in `sync_engine.dart` — that's the tell that the event never fired a second time.

### Pitfall 3: `unawaited(retryFailed())` / `unawaited(processPending())` timing races
**What goes wrong:** Both trigger closures (`start()`'s connectivity listener, `handleResume()`)
call `unawaited(...)` — the returned future is fire-and-forget. A test that asserts DB/API state
immediately after calling `engine.handleResume()` without draining the event queue will
frequently see PRE-drain state (flaky, order-dependent failures).
**Why it happens:** `unawaited` intentionally does not block the caller; the async work
(DB reads/writes, `FakeApiClient.syncTrips`) completes on later microtasks/timers.
**How to avoid:** Always call the existing `_pumpEventQueue()` helper
(`sync_engine_test.dart:634-638`, 20× `Future.delayed(Duration.zero)`) after any trigger call
before asserting — the existing MR-03 test and the one existing gate test already follow this
pattern; extend it, don't skip it.
**Warning signs:** Intermittent CI failures in the new tests that don't reproduce locally on a
fast machine (classic microtask-ordering flake).

### Pitfall 4: In-memory Drift `NativeDatabase.memory()` + `closeStreamsSynchronously: true`
**What goes wrong:** Omitting `closeStreamsSynchronously: true` (or forgetting `await db.close()`
in `tearDown`) can leave the reactive `watchPending()` stream subscription open across tests,
causing state leakage or "Bad state: Cannot add new events after calling close" errors in later
tests within the same file.
**Why it happens:** `AppDatabase` is constructed fresh per test in `setUp`, but Drift's
`NativeDatabase.memory()` streams are not implicitly torn down without an explicit close.
**How to avoid:** Copy the exact existing `setUp`/`tearDown` pair from the "auto-retry time gate"
group (`sync_engine_test.dart:576-610`) — it already has this right; don't hand-write a new
`AppDatabase` construction.
**Warning signs:** Tests pass in isolation (`flutter test test/unit/sync/sync_engine_test.dart -n
"my new test"`) but fail when run as part of the full file/group.

### Pitfall 5: SegmentedButton `find.text('Local')` / `find.text('Cloud')` ambiguity across rows
**What goes wrong:** The merge UI renders a `SegmentedButton<String>` with 'Local'/'Cloud'
labels for EACH of 5 fields (`startTime`, `endTime`, `durationSeconds`, `distanceMeters`,
`direction`). A bare `find.text('Local')` matches 5 widgets once the merge panel is expanded;
`tester.tap()` on an ambiguous finder throws.
**Why it happens:** Flutter's `find.text()` matches by exact string across the whole tree, not
scoped to a Row.
**How to avoid:** The existing test 3 already solves this with
`find.descendant(of: find.ancestor(of: find.text('<fieldName>'), matching: find.byType(Row)).first, matching: find.text('Local'))`
— reuse this exact pattern for `distanceMeters` (or whichever second field is chosen).
**Warning signs:** `Bad state: Too many elements` / `findsOneWidget` failures listing 5 matches
when writing the new D-08 test.

### D-04: consolidated predicate (rename `isAutoRetryExhausted`)
```dart
// Source: lib/sync/sync_engine.dart:102-104 (current, confusingly-named)
/// Whether the auto-retry time gate is open (exhausted).
bool get isAutoRetryExhausted =>
    _lastAutoRetry == null || _now().difference(_lastAutoRetry!) > kFailedAutoRetryWindow;

// Recommended consolidation (exact name is Claude's discretion per CONTEXT.md;
// `autoRetryWindowElapsed` avoids the semantic inversion of "Exhausted" — true
// means "the window has elapsed, an auto-retry may fire now"):
bool get autoRetryWindowElapsed =>
    _lastAutoRetry == null || _now().difference(_lastAutoRetry!) > kFailedAutoRetryWindow;
```

files):

2. `lib/sync/sync_engine.dart:351` — connectivity-restore trigger, inline duplicate condition
   (NOT currently calling the getter — it repeats the raw boolean expression).
3. `lib/sync/sync_engine.dart:368` — `handleResume()`, same inline duplicate.
4. `lib/features/dashboard/screens/dashboard_screen.dart:194` —
   `if (!syncEngine.isAutoRetryExhausted) return const SizedBox.shrink();` inside

After D-04, sites 2 and 3 should call the renamed getter (eliminating the inline duplication),
and site 4 updates its identifier only (behavior unchanged — confirmed no dashboard widget test

### D-05: merge default flip
```dart
// Source: lib/features/settings/widgets/conflict_resolution_sheet.dart:173-176 (current)
selected: {
  _mergeSelections[tripId]?[field] ?? 'cloud',
},

// After D-05:
selected: {
  _mergeSelections[tripId]?[field] ?? 'local',

```
No other leak point found: `_applyAll`'s merge branch (`conflict_resolution_sheet.dart:50`,
`final selections = _mergeSelections[localTrip.id] ?? {};`) reads from the SAME
`_mergeSelections` map the widget writes to — an empty/absent per-field entry falls through to
each field's own `selections['field'] == 'local' ? ... : cloudTrip.field` ternary
(`:54-68`), which already defaults to the CLOUD value whenever the string isn't exactly
`'local'`. This means `_applyAll`'s merge logic has an INDEPENDENT hardcoded 'cloud' fallback
baked into each ternary (`selections['startTime'] == 'local' ? ... : cloudTrip.startTime`) that
is NOT driven by the widget's displayed default — today, an untouched field silently merges to
CLOUD in the database even though the widget hasn't rendered yet (or if `_applyAll` runs via
`Merge All` without ever expanding the panel). **This is a second, distinct leak point** the
D-05 flip must also address, or D-06's accepted "Merge All ≡ Keep All Local" consequence will
NOT actually hold — verify this carefully in planning (see Open Questions).

## Open Questions

1. **Does `_applyAll`'s merge ternary need its own fallback fixed, or does the widget-level
   default alone satisfy D-05/D-06?**
   - What we know: The widget's displayed `SegmentedButton` default (`?? 'cloud'` /
     `?? 'local'`) only controls what's SHOWN and what gets written into `_mergeSelections` when
     the user interacts with a field. `_applyAll`'s merge branch reads `_mergeSelections[id] ??
     {}` and then, per field, `selections['field'] == 'local' ? local : cloud` — an ABSENT key
     (field never touched, panel never expanded, or `Merge All` bulk button used) currently
     always evaluates to the cloud value, regardless of the widget's displayed default.
   - What's unclear: For D-06's accepted "Merge All ≡ Keep All Local when no per-field
     selections" to actually hold post-D-05, `_applyAll`'s absent-selection fallback must ALSO
     become local-preferring — e.g. `(selections['field'] ?? 'local') == 'local' ? local :
     cloud`. Simply flipping line 175's widget default does NOT change `_applyAll`'s behavior for
     the `Merge All` button path, because `Merge All` never opens/renders the per-field panel at
     all — `_mergeSelections[tripId]` stays permanently empty for any trip whose panel was never
     expanded.
   - Recommendation: The planner should treat this as PART of D-05's fix, not a separate item —
     both the widget's displayed default (line 175) AND `_applyAll`'s 5 ternaries (lines 54-68)
     need the same flip, or D-06's accepted consequence (verified by the new D-08 test) will
     fail. Flag this explicitly in the plan's task for D-05 so it isn't a one-line-only change.

2. **Should the D-04 predicate rename also extract the connectivity/resume trigger BODIES into a
   shared private method (not just the boolean condition)?**
   - What we know: CONTEXT.md D-04 asks only to consolidate "the gate condition" (the boolean),
     not the surrounding `if/else` trigger dispatch logic. `handleResume()` is already its own
     method; the connectivity trigger is an inline closure in `start()`.
   - What's unclear: Whether extracting a shared `_fireAutoRetryTrigger()` method (called by both
     the connectivity closure and `handleResume()`) is in scope, versus leaving `handleResume()`
     and the closure as two call sites that both reference the same renamed getter.
   - Recommendation: Minimal-diff approach — keep `handleResume()` as-is (it's already the public
     test seam) and have the connectivity closure call the renamed getter directly, exactly
     mirroring today's structure. Do not extract further; this keeps the diff small per D-01's
     "no reimplementation" spirit and preserves the exact code shape Phase 26 will extend.
