<decisions>
## Implementation Decisions

### Scope reframing
- **D-01:** Accept the existing on-main fixes as the implementation. Phase work =
  add missing tests, apply the small quirk fixes below, and re-verify. Do NOT
  reimplement or deep-audit beyond the named defects.
- **D-02:** Leave Phase 24's `gaps_found` VERIFICATION.md and the stale STATE/ROADMAP
  claims as historical snapshots. This CONTEXT.md documents the discovery; Phase 25.1's
  own verification becomes the current truth. No amendment of old artifacts.

### Auto-retry gate semantics
- **D-03:** Keep the current behavior where a manual Settings "Retry" tap also stamps
  `_lastAutoRetry`, suppressing automatic retries for the following 4h window. Manual
  retry itself is never blocked (user escape hatch preserved). This is now locked-in,
  test-pinned behavior.
- **D-04:** Consolidate the gate condition — currently written out three times
  (`sync_engine.dart:104` getter, `:351` connectivity trigger, `:368` resume trigger) —
  into a single well-named predicate (e.g. `autoRetryWindowElapsed`; exact name is
  Claude's discretion) used by both triggers AND the dashboard banner gate
  (`dashboard_screen.dart:194`). Rename the confusingly-inverted `isAutoRetryExhausted`.

### Merge UX
- **D-05:** Flip the merge per-field default from 'cloud' to 'local'
  (`conflict_resolution_sheet.dart:174-176`). Untouched merge keeps the user's own
  recorded data; fields are explicitly opted INTO cloud values. This makes Merge
  visibly distinct from "Use Cloud" — the original complaint.
- **D-06:** Keep the bulk "Merge All" button (Phase 24 D-04 symmetry). Accepted
  consequence: with local defaults, "Merge All" on trips without explicit per-field
  selections now equals "Keep All Local" — user explicitly accepted this.

### Regression tests
- **D-07:** Auto-retry gate test = **full gate contract**, three assertions using the
  existing injected fake clock (`now:` constructor param):
  1. A second connectivity/resume trigger within `kFailedAutoRetryWindow` does NOT
     re-fire `retryFailed` (no `resetFailed`, no additional API calls).
  2. Advancing the clock past the window DOES re-fire.
  3. A manual `retryFailed()` stamps the window so a following auto trigger within the
     window is suppressed (pins D-03).
- **D-08:** Strengthen the existing merge widget test
  (`test/widget/features/settings/conflict_resolution_sheet_test.dart`): use a conflict
  where local and cloud differ on TWO fields; select one as local, leave one as cloud;
  assert the merged row differs from BOTH pure Use Cloud AND pure Keep Local outcomes.
  Update test expectations for the D-05 default flip (defaults are now 'local').

### Claude's Discretion
- Exact name of the consolidated gate predicate (must not read inverted).
- Test placement/organization within the existing `sync_engine_test.dart` "auto-retry
  time gate" group and the existing conflict-sheet widget test file.
- Any fixture helpers needed for the two-differing-fields merge conflict.
- Whether the dashboard banner gate needs its own distinct predicate if semantics
  diverge from the trigger gate during implementation (keep behavior identical either way).

</decisions>

<deferred>
## Deferred Ideas

- **Extract merge logic into a pure function** — rejected for this phase (ship-fast),
  but Phase 26 extends the merge with breaks/edit-metadata fields; a testable seam is
  a natural Phase 26 refactor. Note in Phase 26 planning.
- **Overlap-conflict UUID semantics** — "Use Cloud"/merge on an `OverlapConflict`
  writes cloud fields into the LOCAL trip's UUID row; the cloud trip's own UUID never
  lands locally. Works, but means the cloud copy remains in Firestore under a different
  id and could re-conflict on future restores. Revisit if users report repeat conflicts.
- **"Highlight differing fields only" merge UI** — considered for D-05, rejected for
  scope; candidate polish item if the merge sheet gets attention later.
