## Scope of This Contract

This is a bug-fix phase with a deliberately tiny UI delta. **No new screens, no new
components, no palette/typography changes.** The entire visible surface is the existing
`ConflictResolutionSheet` (`lib/features/settings/widgets/conflict_resolution_sheet.dart`),
already specified in `.planning/phases/24-automatic-cloud-sync-restore/24-UI-SPEC.md`
(approved 2026-06-16). This document:

1. **Re-affirms** Phase 24's design system, typography, color, and copy for the sheet —
   nothing here changes them.
2. **Locks the one visible delta**: the per-field merge `SegmentedButton` default flips
   from `'cloud'` to `'local'` (D-05), at BOTH the displayed-default call site and the
   `_applyAll` fallback ternaries (RESEARCH.md's Open Question #1 — both must move
   together or the visible default and the applied result diverge).
3. **Locks the unchanged-surface contract**: bulk buttons ("Keep All Local" / "Use All
   Cloud" / "Merge All") keep their exact current labels, order, and styling (D-06).
4. **Confirms zero UI change** for the auto-retry throttle fix (D-04) — the dashboard
   stuck-item banner (`SyncStuckBanner`, from Phase 24's spec) must render identically
   before and after the predicate rename.

There is no shadcn gate for this phase — this is a Flutter/Dart project (Material 3 +
Traevy `ThemeExtension` tokens), not React/Next/Vite. `components.json` does not apply.

## Interaction Contract

### Per-field merge default (D-05 — the phase's one visible interaction change)

**Before this phase:** On expanding "Merge" for a conflicted trip, every per-field
`SegmentedButton` (`startTime`, `endTime`, `durationSeconds`, `distanceMeters`, `direction`)
renders with **"Cloud" pre-selected** (accent-filled). An untouched field silently resolves
to the cloud value on submit.

**After this phase:** Every per-field `SegmentedButton` renders with **"Local"
pre-selected** (accent-filled) instead. An untouched field silently resolves to the
**local** value on submit. The user must explicitly tap "Cloud" per field to opt a field
INTO the cloud value.

**Two-location requirement (do not implement as a single-line change):**
1. **Displayed default** — `conflict_resolution_sheet.dart:~175`, the `SegmentedButton`'s
   `selected: {_mergeSelections[tripId]?[field] ?? 'cloud'}` → `?? 'local'`. Controls what
   the user SEES pre-highlighted.
2. **Applied fallback** — `conflict_resolution_sheet.dart:~54-68`, `_applyAll`'s five
   per-field ternaries (`selections['startTime'] == 'local' ? ... : cloudTrip.startTime`,
   and the same shape for `endTime`, `durationSeconds`, `distanceMeters`, `direction`).
   These currently fall through to the CLOUD value whenever `selections[field]` is absent
   (panel never expanded, or the "Merge All" bulk button used without ever opening a
   per-trip panel) — INDEPENDENT of the widget's displayed default. Each ternary must
   become local-preferring, e.g. `(selections['startTime'] ?? 'local') == 'local' ? ... :
   cloudTrip.startTime`, so that what's applied always matches what's (or would be)
   displayed as selected.

**Why both matter to this contract:** If only the displayed default (#1) flips, a user who
never expands the per-field panel — including anyone who taps "Merge All" in bulk — gets a
result that visually promised "Local" but silently wrote "Cloud" to Drift. That would
reopen exactly the "Merge looks different but behaves the same" complaint this phase exists
to fix. The UI contract is: **what's shown selected is what gets written**, with no hidden
divergent fallback.
