> ⚠️ **CRITICAL PATTERN CORRECTION (read before planning):** RESEARCH §1/§3 and the
> `pattern_mapping_context` both say `AuthStateNotifier extends StateNotifier<AuthState>` with a
> `StateNotifierProvider`. **The codebase does NOT use `StateNotifier`.** Every stateful notifier in
> this repo uses the Riverpod 3.x **`Notifier<T>` + `NotifierProvider<N, T>`** form
> (`TrackingNotifier extends Notifier<TrackingState>`, `TripManagementNotifier extends
> Notifier<TripManagementState>`). `grep "StateNotifier" lib/` returns **zero hits**. The planner MUST
> implement `AuthStateNotifier extends Notifier<AuthState>` to match the codebase, NOT the
> `StateNotifier` shown in research. This also changes the subscription lifecycle: subscribe inside
> `build()` and cancel via `ref.onDispose(...)` (see `TrackingNotifier`), not a constructor + manual
> `dispose()`. RESEARCH Assumption A5 already flags `.when()` as wrong; this is the matching base-class
> correction.

### `lib/features/auth/models/auth_state.dart` (model, sealed state)

**Analog:** `lib/features/tracking/state/tracking_state.dart` (also `trip_management_providers.dart` lines 14-42)

**Copy the sealed-class shape exactly** — `@immutable sealed class` base with `final class` subtypes, `const` constructors for singletons, a constructor-validated payload subtype. From `tracking_state.dart` lines 25-105:

```dart
@immutable
sealed class TrackingState {
  const TrackingState();
}

final class TrackingIdle extends TrackingState {
  const TrackingIdle();          // singleton const, identical() at call sites
}

final class TrackingActive extends TrackingState {
  const TrackingActive({ required this.startedAt, /* ...payload fields... */ });
  final DateTime startedAt;      // payload subtype carries final fields
}

final class TrackingError extends TrackingState {
  TrackingError(this.message) {  // constructor validates non-empty invariant
    if (message.isEmpty) { throw ArgumentError.value(message, 'message', '...'); }
  }
  final String message;
}
```

**Apply to AuthState:** `AuthLoading` (const singleton, like `TrackingIdle`), `AuthGuest` (const
singleton), `AuthSignedIn({required uid, name, email})` (payload subtype, like `TrackingActive`).
Match the doc-comment convention ("Use exhaustive switch... Never add a default branch" — see
`trip_management_providers.dart` line 13).

### `lib/features/auth/providers/auth_providers.dart` (provider, event-driven)

**Analog:** `lib/features/tracking/providers/tracking_providers.dart` (the `TrackingNotifier` + its providers)

**Notifier base class + stream subscription in `build()`** (lines 94-147) — this is the load-bearing
pattern. `AuthStateNotifier` subscribes to `FirebaseAuth.instance.authStateChanges()` the same way
`TrackingNotifier` subscribes to fbs events:

```dart
final NotifierProvider<TrackingNotifier, TrackingState> trackingStateProvider =
    NotifierProvider<TrackingNotifier, TrackingState>(
      TrackingNotifier.new,
      name: 'trackingStateProvider',           // every provider sets `name:`
    );

class TrackingNotifier extends Notifier<TrackingState> {
  StreamSubscription<Map<String, dynamic>?>? _stateSub;

  @override
  TrackingState build() {
    ref.onDispose(() {                          // cancel in onDispose, NOT dispose()
      unawaited(_stateSub?.cancel());
    });
    _attach();                                   // subscribe here
    return const TrackingIdle();                 // initial state = AuthLoading()
  }

  void _attach() {
    _stateSub = service.on(kEvent).listen((data) {
      state = next;                              // assign `state =` to emit
    }, onError: (Object error, StackTrace stack) { /* ... */ });
  }
}
```

**Apply to auth:** `build()` returns `const AuthLoading()`; check the injected `firebaseReady` flag
(D-15 degrade path — set `state = const AuthGuest()` and return early if false); else subscribe to
`authStateChanges()` and map `user == null ? AuthGuest() : AuthSignedIn(...)`. Cancel the
`StreamSubscription<User?>` in `ref.onDispose`.

**Plain-service providers** (lines 51-69) — copy verbatim for `firebaseAuthProvider`,
`googleSignInProvider`, `secureStorageProvider`, `authServiceProvider` so tests can override them
(RESEARCH §Wave 0 demands these be injectable):

```dart
final Provider<TrackingPermissionService> trackingPermissionServiceProvider =
    Provider<TrackingPermissionService>(
      (ref) => const TrackingPermissionService(),
      name: 'trackingPermissionServiceProvider',
    );
```

**`firebaseReadyProvider` override pattern** — `main.dart` injects the flag via
`overrideWithValue`. See the `notificationServiceProvider` override note in `settings_providers.dart`
lines 24-33 ("Widget tests override this with a fake to avoid platform-channel calls") — same
rationale: `FirebaseAuth.instance` / `GoogleSignIn.instance` crash on the test host, so they MUST be
provider-injected.

**Manual-provider rationale comment** — copy the canonical block from `lib/database/providers.dart`
lines 7-29 (no `@riverpod`, analyzer-version conflict, bare `Provider(...)` = keepAlive). Reference
it, don't re-derive it.

rows). `backfillUserId` returns the changed-row count (see DAO below); `return changed > 0`.

### `lib/database/daos/trips_dao.dart` (MODIFY — add `backfillUserId`) (dao, CRUD batch update)

**Analog:** the existing `updateTrip` method in the SAME file (lines 115-119)

**Copy the explicit-WHERE update pattern verbatim** — STATE.md / CLAUDE.md ban `.replace()` for
partial updates (RESEARCH Anti-Patterns):

```dart
/// Update the trip identified by `companion.id.value`. Only columns
/// wrapped in `Value(...)` are touched...
/// Pitfall 4 mitigation: the WHERE clause is set explicitly...
/// Never use `update(trips).replace(row)` for partial updates.
Future<void> updateTrip(TripsCompanion companion) {
  return (update(
    trips,
  )..where((t) => t.id.equals(companion.id.value))).write(companion);
}
```

**New method** — same `(update(trips)..where(...)).write(...)` form, filter on `userId.equals(kDefaultUserId)`,
**return the changed count** so the service can detect first-sign-in (RESEARCH §2 example).
`(update(trips)..where((t) => t.userId.equals(kDefaultUserId))).write(TripsCompanion(userId: Value(newUserId)))`
returns `Future<int>`. Match the doc-comment style (Pitfall reference + "Never use .replace()").

> The DAO is scoped narrowly by convention (lines 64-69: "adding [methods] before then would be dead
> code"). `backfillUserId` is justified now because Phase 9 calls it — no speculative variants.

## Shared Patterns

### Manual Riverpod providers (no codegen)
**Source:** `lib/database/providers.dart` lines 7-45
**Apply to:** `auth_providers.dart` (all providers), the `_AccountSection` Consumer conversion
Bare `Provider(...)` / `NotifierProvider(...)` = `isAutoDispose = false` = keepAlive. Every provider
sets a `name:`. No `@riverpod` annotations. **Use `Notifier<T>`, never `StateNotifier<T>`** (see the
CRITICAL CORRECTION at top).

### Sealed state + exhaustive switch (no default)
**Source:** `lib/features/tracking/state/tracking_state.dart` lines 25-105;
`lib/features/trips/providers/trip_management_providers.dart` lines 14-42
**Apply to:** `auth_state.dart`, the `app.dart` gate, the `_AccountSection` render
`@immutable sealed class` + `final class` subtypes; `const` singletons; `switch` with no `default`
branch so a new variant is a compile error.

### Transaction-wrapped DAO orchestration
**Source:** `trip_management_providers.dart` lines 64-83 (`editTrip`)
**Apply to:** `AuthService.signIn()` backfill of trips + user_preferences
`await db.transaction(() async { ...two DAO writes... })`. Build any payload/read BEFORE the mutating
call (Pitfall 3 discipline, lines 100-108).

### Explicit-WHERE updates (never `.replace()`)
**Source:** `trips_dao.dart` `updateTrip` lines 115-119
**Apply to:** both `backfillUserId` methods
`(update(table)..where((t) => t.col.equals(x))).write(Companion(...))`. STATE.md / CLAUDE.md mandate.

### Provider injection for testability (override seam)
**Source:** `settings_providers.dart` lines 24-33 (`notificationServiceProvider` override note);
`tracking_providers.dart` lines 78-89 (ref.watch dependency injection)
**Apply to:** `firebaseAuthProvider`, `googleSignInProvider`, `secureStorageProvider`,
`firebaseReadyProvider`, `authServiceProvider`
FlutterFire/GoogleSignIn singletons crash on the test host — they MUST be reachable only via
overridable providers (RESEARCH Wave 0).

### `context.mounted` guard after every await
**Source:** `lib/features/trips/services/trip_actions.dart` lines 42, 48
**Apply to:** onboarding onTap handler, sign-in sheet handler (any post-`signIn()` navigation)

### Token / async-boundary discipline (PII + ordering)
**Source:** `tracking_providers.dart` lines 190-203 (PII guard); `trip_management_providers.dart`
lines 100-108 (build-before-mutate)
**Apply to:** `AuthService` — never log `idToken`; `await` backfill before navigating (Pitfall 7).

### Traevy token + font access
**Source:** `onboarding_screen.dart` line 21; `account_row.dart` lines 37, 43-60; `traevy_logo_mark.dart`
**Apply to:** splash, success screen, sign-in sheet
`Theme.of(context).extension<TraevyTokensExt>()!` for `bg`/`accent`/`accentBg`/`textDim`/`borderStr`;
`TraevyFonts.ui(...)` / `TraevyFonts.mono(...)` for text. Reuse `GoogleContinueButton` shell; do not
re-style.
