```dart
// lib/features/tracking/services/tracking_notification_service.dart

/// UX-03 foreground notification manager. Owns the single "Active commute"
/// channel and the single ongoing notification shown while the service
/// isolate is running.
///
/// This service is UI-isolate only — the background notification response
/// handler is a top-level function (see `trackingNotificationBackgroundHandler`
/// below) per flutter_local_notifications' `@pragma('vm:entry-point')` rule.
class TrackingNotificationService {
  TrackingNotificationService({
    FlutterLocalNotificationsPlugin? plugin,
  }) : _plugin = plugin ?? FlutterLocalNotificationsPlugin();

  final FlutterLocalNotificationsPlugin _plugin;

  /// Initialise the plugin with the notification channel and the foreground
  /// / background response handlers. Called ONCE from main() before
  /// runApp(). Idempotent — safe to call twice.
  Future<void> initialize();

  /// Show the static "Recording commute" notification with the Stop action
  /// button. Called from the service isolate entry or from the UI isolate
  /// at service-start time (Phase 2 calls it from the UI isolate in
  /// TrackingServiceController.start so the notification is bound to the
  /// UI isolate's response handler).
  Future<void> showRecording();

  /// Cancel the notification. Called from the finalize path (both save
  /// and discard — Pitfall 3).
  Future<void> dismiss();
}

/// Top-level background response handler. Required to be top-level AND
/// annotated with @pragma('vm:entry-point') so the VM doesn't tree-shake
/// it in release builds (Pitfall 4).
@pragma('vm:entry-point')
void trackingNotificationBackgroundHandler(NotificationResponse response);
```

```dart
      }
    }
```

|----------|-------------|
| Notification response payload (OS → app) | OS may replay or deliver unknown action ids |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-17 | Tampering | Notification response handler receives an unknown actionId | mitigate | Both foreground and background handlers check `response.actionId == kTrackingStopActionId` exactly. Unknown ids are silently ignored. (V5 validation.) |
| T-02-18 | Information Disclosure | Notification body text could leak trip details | mitigate | D-14 locks the body to `null` (title only, static "Recording commute"). No per-sample updates. |
| T-02-19 | Tampering | A crafted trip_finalized payload could inject bad data into Drift | mitigate | `FinalizedTrip.fromMap` (from plan 02-02) does typed casts on every field. persistFinalizedTrip then validates duration/distance thresholds before insertion. |
| T-02-20 | Denial of Service | persistFinalizedTrip failure leaves notification visible | mitigate | `dismiss()` is called in EVERY exit path of persistFinalizedTrip — success, discard, and catch block (Task 2 implementation). |
| T-02-21 | Elevation of Privilege | Background response handler tree-shaken in release | mitigate | `@pragma('vm:entry-point')` on `trackingNotificationBackgroundHandler` (Task 1). Deviation rules explicitly flag removal as forbidden. |
| T-02-22 | Information Disclosure | Error message in PersistFailed snackbar leaks stack trace | mitigate | Task 3 formats the snackbar as `'Unable to save trip: ${error}'` — the `.toString()` of the caught object. Audit: do NOT include `StackTrace.current` or raw Drift SQL strings. |
