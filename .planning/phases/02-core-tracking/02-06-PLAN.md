    Hand-rolled TestNotifier lives as a private class at the top of the test file:

    ```dart
    class _TestTrackingNotifier extends TrackingNotifier {
      int startCallCount = 0;

      @override
      TrackingState build() {
        // Skip the service.on subscription — tests drive state directly.
        return const TrackingIdle();
      }

      @override
      Future<void> start() async {
        startCallCount++;
        state = const TrackingStarting();
      }

      void setState(TrackingState next) => state = next;

      void simulateDiscard() {
        // Inject the result the same way the real listener does.
        // Test helper only — production code does this in the
        // trip_finalized listener (plan 02-05 Task 2).
        state = const TrackingStopping();
        // Uses the @visibleForTesting seam added by plan 02-05 Task 2
        // (`setLastPersistResultForTesting`). Plan 02-06 does NOT add it —
        // it calls it. Inherited on this TestNotifier subclass via super.
        setLastPersistResultForTesting(const PersistDiscardedTooShort());
        state = const TrackingIdle();
      }
    }
    ```

That seam is NOT added by this plan — it already exists, added by **plan 02-05 Task 2** in the same commit that introduces `_lastPersistResult`. <task type="auto">
  <name>Task 2: HomeScreen widget test — permission dialog path</name>
  <files>test/widget/features/tracking/home_screen_test.dart</files>
  <action>
    Create `test/widget/features/tracking/home_screen_test.dart`. Override `trackingPermissionServiceProvider` with a real `TrackingPermissionService` constructed via its `@visibleForTesting` named constructor `TrackingPermissionService.forTesting(...)` (added in plan 02-01 Task 3). Do NOT use `implements TrackingPermissionService` — the class is concrete, not an interface, and very_good_analysis flags `implements` on a non-sealed concrete class.

    Use a small harness that wraps closures for `probe`, `requester`, and `opener`:

    ```dart
    /// Builds a TrackingPermissionService whose probe/request both return
    /// [status], with a spy counter for openSystemSettings calls.
    ({TrackingPermissionService service, int Function() openSettingsCalls})
        _buildFakePermissionService(TrackingPermissionStatus status) {
      var openCount = 0;
      // Map the high-level TrackingPermissionStatus back to the
      // permission_handler PermissionStatus the real class expects from the
      // probe/requester closures. The forTesting constructor accepts raw
      // PermissionStatus values so tests can short-circuit the two-step flow.
      final mapped = switch (status) {
        TrackingPermissionStatus.fullyGranted => PermissionStatus.granted,
        TrackingPermissionStatus.foregroundOnly => PermissionStatus.granted,
        TrackingPermissionStatus.denied => PermissionStatus.denied,
        TrackingPermissionStatus.permanentlyDenied =>
          PermissionStatus.permanentlyDenied,
      };
      final service = TrackingPermissionService.forTesting(
        probe: (_) async => mapped,
        requester: (_) async => mapped,
        opener: () async {
          openCount++;
          return true;
        },
      );
      return (service: service, openSettingsCalls: () => openCount);
    }
    ```

    (If the mapping from `TrackingPermissionStatus` back to `PermissionStatus` is too lossy for a given test — e.g. distinguishing `foregroundOnly` from `fullyGranted` requires two different probes for when-in-use vs always — drive the two closures independently. The harness above is the default; individual tests may hand-build a service with custom closures.)

    Four tests:

    1. **Start commute button is visible.** Pump HomeScreen with `fullyGranted`. Assert `find.widgetWithText(ElevatedButton, 'Start commute')` present.

    2. **Tap with `fullyGranted` navigates to /tracking.** Use a `MaterialApp(routes: kAppRoutes, home: HomeScreen())`. Tap Start. Pump a few frames. Assert `find.byType(TrackingScreen)` is present in the tree (the route was pushed).

    3. **Tap with `permanentlyDenied` shows the settings dialog instead of navigating.** Override the fake with `permanentlyDenied`. Tap Start. `await tester.pump()`. Assert `find.text('Open settings')` (dialog button) is present AND `find.byType(TrackingScreen)` is NOT present.

    4. **Tapping 'Open settings' in the dialog calls openSystemSettings.** Continue test 3: build the harness via `_buildFakePermissionService(TrackingPermissionStatus.permanentlyDenied)`, override `trackingPermissionServiceProvider` with `harness.service`, tap the 'Open settings' dialog button, assert `harness.openSettingsCalls() == 1`.

    Run `flutter test test/widget/features/tracking/home_screen_test.dart` — all four pass.

<deviation_rules>
- Rule 1 (`@visibleForTesting` seam): The `setLastPersistResultForTesting` helper is the ONLY production-code mutation in this plan. Document it inline and flag it for reviewer attention. Do NOT expose `_lastPersistResult` directly as a public setter — the seam is explicitly test-only.
- Rule 2 (TestNotifier subclass vs Notifier.new override): Riverpod 3.x allows `NotifierProvider.overrideWith(NotifierNew)` — the notifier constructor tear-off. The test subclass must provide a no-argument constructor tear-off via a top-level `_TestTrackingNotifier.new`. If the notifier constructor requires parameters in later plans, switch to `overrideWith(() => _TestTrackingNotifier())`.
- Rule 3 (skipped checkpoint): Accepting `skipped:` on Task 3 is a real deviation that the user explicitly acknowledges. Record in the phase SUMMARY that device verification was not performed, and open a follow-up tracking item.
- Auto-fix allowed: test helper naming, finder selector alternatives (Key vs text), doc comment wording.
- Halt and escalate: any widget test that requires platform channel mocking beyond what flutter_test provides by default, any case where Riverpod 3.x notifier override semantics differ from what the test assumes.
</deviation_rules>
