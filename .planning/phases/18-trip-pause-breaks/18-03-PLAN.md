});

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| UI isolate ↔ service isolate | Pause/Resume commands cross flutter_background_service.invoke. The payload must be a primitive command name only — a non-primitive would fail to serialize and the command would silently no-op. |
| UI state ↔ source of truth | The accumulator (service isolate) is the only authority on paused state. If the UI ran its own pause clock it could drift from the persisted record after a kill/restart. |
| notification action ↔ command channel | (Plan 18-04 reuses) the same pause command path — a stale/spoofed action id must not toggle pause unexpectedly. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-08 | Spoofing | UI↔service pause/resume command | mitigate | The service handler matches the exact event name (kTrackingPauseCommand/kTrackingResumeCommand) and ignores everything else, mirroring the Stop handler. No payload is parsed — only the channel name routes the command. |
| T-18-09 | Tampering | UI drifting from the true paused state | mitigate | UI is a dumb terminal (D-08) — it renders snapshot.isPaused with no local pause timer. After backgrounding/kill the UI reconnects and the first snapshot dictates state, so the displayed state can never diverge from the accumulator. A widget test asserts the paused UI is driven purely by the injected snapshot. |
| T-18-10 | Denial of service | pause command racing Stop | mitigate | The pause/resume handlers early-return when `stopping` is true, so a late pause cannot touch a finalized accumulator (mirrors the stop-race guard). |
</threat_model>
