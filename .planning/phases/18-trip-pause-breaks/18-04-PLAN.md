<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| detector ↔ stuck-time metric | The detector must consume the SAME stuck classification as the accumulator — a second speed threshold would risk diverging from (and corrupting confidence in) the core stuck-time metric. |
| notification action ↔ pause command | The Pause action posts across isolates to kTrackingPauseCommand; a stale/spoofed action id must not toggle pause. |
| preference ↔ prompt gate | auto_pause_enabled is the opt-in gate; if the prompt fired regardless of the flag, the feature would not be opt-in (SC#5 violation). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-11 | Tampering | false-positive stationary detection corrupting trust in stuck-time | mitigate | Detection keys off continuous STUCK intervals from the accumulator's own classification (D-11); any moving interval resets the streak. A test proves stop-and-go never reaches threshold. No raw-speed path and no second threshold constant exist. |
| T-18-12 | Spoofing | spoofed/stale Pause action id | mitigate | Both notification handlers match the exact kTrackingAutoPauseActionId and ignore everything else (mirrors the Stop V5 validation, T-02-17); only then invoke kTrackingPauseCommand. |
| T-18-13 | Elevation of privilege | prompt firing while opt-out | mitigate | The prompt post is gated on `autoPauseEnabled && !isPaused && shouldPrompt()`; default false (18-01 column default + settings default). A settings test asserts default OFF; the gate ensures OFF → no post (SC#5). |
| T-18-14 | Denial of service | repeated prompts spamming the shade | mitigate | shouldPrompt() latches once per streak and only re-arms after onMovingInterval(); at most one prompt per stationary streak (D-12). |
</threat_model>
