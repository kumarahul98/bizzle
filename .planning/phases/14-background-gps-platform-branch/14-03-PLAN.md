Current TrackingNotifier._attach() (tracking_providers.dart lines 149-262) subscribes directly to
`FlutterBackgroundService()` via four `service.on(<event>)` calls:
  _readySub  ← kServiceReadyEvent  → re-post notification (Android only)
  _stateSub  ← kTrackingStateEvent → trackingActiveFromSnapshotMap(data) → state
  _finalizeSub ← kTripFinalizedEvent → FinalizedTripCodec.fromEventMap → persistFinalizedTrip → idle
  _errorSub  ← kTrackingErrorEvent  → switch on data['reason'] → TrackingError(message)
All four subscriptions are cancelled in ref.onDispose and by _cancelSiblingSubs(). The decoders
(trackingActiveFromSnapshotMap, FinalizedTripCodec) operate on the SAME map shapes the iOS engine emits,
so ONLY the stream source changes — the decode/persist/error logic stays put.

Existing providers in this file:
  trackingServiceControllerProvider : Provider<TrackingServiceController>
  trackingStateProvider : NotifierProvider<TrackingNotifier, TrackingState>
Manual providers only (Phase 1 D-12 — no @riverpod codegen); bare Provider/NotifierProvider = keepAlive.

Plan 02 artifacts:
  TrackingEventSource { onState, onFinalized, onError, onReady, start(), stop() }
  FbsTrackingEventSource(FlutterBackgroundService service)
  MainIsolateTrackingEngine()  // implements TrackingEventSource, onReady = const Stream.empty()

The controller already issues start/stop; the iOS gate (Plan 02) makes controller.start() return false
when accuracy is blocked. TrackingNotifier.start() already maps a false start to
`TrackingError('Unable to start tracking')` (lines 341-357). For IOS-08, the message should be a stable
distinct string (NOT the position_stream_error reason) so the user understands accuracy was the cause.
