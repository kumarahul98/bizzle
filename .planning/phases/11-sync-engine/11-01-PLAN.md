Backend zod tripSchema (backend/functions/src/utils/validation.ts) — the serializer
output MUST satisfy this exactly:
```ts
tripSchema = z.object({
  id: z.string().uuid(),
  userId: z.string().optional(),          // OMIT from client payload — server forces it
  startTime: z.string().datetime(),       // ISO-8601 with Z
  endTime: z.string().datetime(),         // ISO-8601 with Z
  durationSeconds: z.number().int().nonnegative(),   // number, 0 not null
  distanceMeters: z.number().nonnegative(),          // number, 0 not null
  routePolyline: z.string().max(100000).nullable(),  // null allowed
  direction: z.enum(['to_office', 'to_home']),       // stored string as-is
  timeMovingSeconds: z.number().int().nonnegative(),
  timeStuckSeconds: z.number().int().nonnegative(),
  isManualEntry: z.boolean(),
  createdAt: z.string().datetime(),
  updatedAt: z.string().datetime(),
});
// POST body: { trips: tripSchema[] } (min 1, max 1000)
```

Restore response shape (restore-trips.ts):
`200 { statusCode: 200, body: { data: { trips: Trip[] } } }`
where each Trip has the camelCase fields above PLUS `userId` (which the client
ignores on insert — Drift userId is set by auth backfill, not the server copy).

Sync response shape (sync-trips.ts):
`200 { statusCode: 200, body: { data: { syncedIds: string[] } } }`
Error shape (all handlers): non-2xx with `{ statusCode, body: { error: string } }`.

Drift `TripRow` fields (lib/database/tables/trips_table.dart) the serializer reads:
`id:String, userId:String, startTime:DateTime(UTC), endTime:DateTime(UTC),
durationSeconds:int, distanceMeters:double, routePolyline:String?, direction:String,
timeMovingSeconds:int, timeStuckSeconds:int, isManualEntry:bool, createdAt:DateTime,
updatedAt:DateTime`. Manual entries store 0 for the numeric fields (non-nullable

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Flutter client → HTTPS Cloud Functions | Outbound sync payload + Bearer token cross the network to the deployed Phase 10 API. |
| Server JSON → Drift (restore) | Untrusted server response body parsed and inserted into the local DB. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Information disclosure | api_client.dart / SyncException | mitigate | `SyncException.toString()` and all error surfaces NEVER include the Bearer token, uid, or email — only `statusCode` and a short message. Tests assert no token leakage in thrown messages. |
| T-11-02 | Spoofing | api_client.dart token attach | mitigate | Always attach a FRESH `getIdToken()`; on 401 force-refresh once (`getIdToken(true)`). `currentUser == null` ⇒ throw `notSignedIn`, never send an unauthenticated request (D-03). |
| T-11-03 | Tampering | restore parser (server JSON → Drift) | mitigate | `fromJson` constructs a typed `TripsCompanion` with explicit casts (`as num/.toDouble()`, `as int`, `as bool`, `DateTime.parse`); a malformed field throws during parse rather than corrupting a row. `userId` from the server is ignored (not written) — local auth owns it. |
| T-11-04 | Denial of service | syncTrips batch size | accept | Batch chunking (≤1000) and retry/backoff live in Plan 02 (engine); the transport sends what it's given. Server enforces the 1000 cap (M1) and rejects oversized batches with 400 → surfaces as `SyncException`. |
| T-11-05 | Repudiation | n/a (one-way client-authoritative push) | accept | No server→client authority; client is sole writer. No audit requirement in v0.1 scope. |
</threat_model>
