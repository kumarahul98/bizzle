<objective>
Build the client-side sync engine that drains the Drift `sync_queue` to the Phase 10
backend one-way, in the background, with retries + exponential backoff, and wire it to
fire on the three D-07 triggers (post-save, connectivity-restored, app-resume). The
engine is eagerly mounted in `app.dart` without blocking the UI build.

Purpose: Satisfies SYNC-02 and Phase 11 success criteria 1 (queue processed in background
when online), 2 (3 retries with exponential backoff), and 4 (sync never blocks UI). This
is the processing core; the transport (`ApiClient` + `SyncException.retryable`), status
model (`SyncStatus` / `syncStatusProvider`), serialization (`TripSerializer.toJson`), and
DAO `markFailed`/`resetFailed` are delivered by Plan 01 (Wave 1) and IMPORTED here — not
redefined. This plan adds the per-tripId queue collapse (HIGH-1), retryable-aware failure
branching (HIGH-2), and the backoff-window trigger guard (MEDIUM-2).

Output: `lib/sync/sync_engine.dart` (`SyncEngine` + `syncEngineProvider`), three new
constants in `constants.dart`, `connectivity_plus` dependency, an eager mount in
`app.dart`, and a green unit suite under `test/unit/sync/` that exercises every branch
with NO network and NO real connectivity/platform channels.

## Public surface Plans 03 binds to (DO NOT RENAME) — resolves plan-check H2
The engine is exposed as a PLAIN `Provider<SyncEngine>` (a Riverpod `Provider` holding a
service instance) — it is NOT a `Notifier`/`NotifierProvider`, so it has NO `.notifier`.
Plan 03's Settings "tap to retry" row binds to these exact members on the instance:
- `Future<void> SyncEngine.processPending()` — the queue drain. Internally it (1)
  COLLAPSES the pending queue to one effective op per tripId (HIGH-1), (2) branches on
  `SyncException.retryable` for each failure (HIGH-2), and (3) honors an active backoff
  window so triggers don't bypass the timer (MEDIUM-2).
- `Future<void> SyncEngine.retryFailed()` — the explicit retry entrypoint: it FIRST clears
  the backoff window (MEDIUM-2), then `await syncQueueDao.resetFailed(); await processPending();`.
  This is the ONLY method Plan 03's "Sync failed — tap to retry" row calls, via
  `ref.read(syncEngineProvider).retryFailed()` (NO `.notifier`). The contract is explicit
  here so Plan 03 does not fall back to calling resetFailed()/processPending() separately.
- `Duration SyncEngine.backoffDelay(int retryCount)` — pure, public (for tests).
- `final Provider<SyncEngine> syncEngineProvider` — keepAlive, eagerly mounted in app.dart.
