<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pub.dev registry → local machine | Package downloads during `flutter pub get` cross a trust boundary; compromised dependency = compromised build |
| Developer machine → Android device | Debug APK installation on a physical device crosses a trust boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | pubspec.lock transitive dependencies | mitigate | Caret-range pinning on all direct deps (drift ^2.32.1, flutter_riverpod ^3.3.1, etc). Commit `pubspec.lock` to git so future `flutter pub get` reproduces the exact tree. Verify EOL `sqlite3_flutter_libs` is absent post-install. |
| T-01-02 | Information Disclosure | Android debug keystore | accept | Default debug keystore is Flutter-standard and unsuitable for production. Phase 1 only builds debug APKs; release signing config arrives in a later polish phase. |
| T-01-03 | Elevation of Privilege | Android minSdk 34 attack surface | accept | User explicitly accepted API 34 baseline in D-08. Android 14 has newer security model than older APIs — this is actually a mitigation, not a threat. |
| T-01-04 | Spoofing | Package name `traevy.traevy` collision on Google Play | accept | Phase 1 is local development only; Play Store registration happens at release time. If the package name is taken, it's trivially changed before publishing. |
</threat_model>
