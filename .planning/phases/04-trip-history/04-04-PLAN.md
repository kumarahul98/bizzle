<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Route argument ↔ TripDetailScreen | tripId String from Navigator arguments — used as DB query parameter |
| TripDetailScreen ↔ OSM tile server | Map tile HTTP requests made at runtime (network, not user input) |
| AppBar delete ↔ handleDeleteTrip | Async gap: user may navigate away during delete confirmation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Spoofing | tripId route argument | accept | tripId is a UUID-shaped String sourced from within the app (TripCard pushes it from a Drift-fetched TripSummary). findById returns null for non-existent IDs — screen shows 'Trip not found' gracefully. No external attacker can supply route arguments in a native Android app. |
| T-04-04-02 | Information Disclosure | OSM tile requests | accept | Tile requests reveal approximate viewport bounds (centered on the commute route) to OpenStreetMap CDN. This is a known tradeoff of using OSM tiles in a route-display app. No auth credentials, no PII, no trip details in the tile URL. Disclosed in app privacy policy. |
| T-04-04-03 | Denial of Service | decodedToLatLng with very long polyline | accept | Polylines are stored from Tracelet GPS output — bounded by trip duration (~30 min × 1 sample/3s = ~600 points). Memory impact is negligible. No external user-supplied polyline input. |
| T-04-04-04 | Tampering | Navigator.pop after delete | mitigate | context.mounted guard after handleDeleteTrip returns before calling Navigator.pop. TripManagementSaved state is checked before pop — ensures pop only happens on confirmed delete success, not on cancel or error. |
</threat_model>
