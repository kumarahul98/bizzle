# Phase 15 — UI Design Contract

> Visual and interaction contract for Phase 15: Notifications, Permissions & Onboarding UX on iOS.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Important:** This is a Flutter (Material 3 / Traevy design system) + native SwiftUI project.
> The "shadcn" and "registry" sections do not apply — Flutter uses the Traevy token system
> established in Phase 8. The native iOS Widget Extension (ActivityKit) uses SwiftUI and
> follows Apple HIG, with token values translated into the equivalent SwiftUI color/font calls.

## Spacing Scale

Standard Traevy 8-point scale. All values are multiples of 4. Carry-forward from Phase 8.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, inline chips |
| sm | 8px | Compact element spacing, intra-row gaps |
| md | 16px | Default element and section padding, FeatureTick inter-row gap |
| lg | 24px | Card internal padding, section vertical gaps |
| xl | 32px | Screen horizontal padding (onboarding matches existing 32px) |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level vertical spacing |

**Exceptions:**
- Onboarding screen top padding: 60px (matches existing `fromLTRB(32, 60, 32, 32)`)
- PermissionBanner: zero padding — MaterialBanner renders its own insets
- iOS Live Activity lock-screen layout: follows Apple HIG WidgetKit sizing constraints
  (compact leading/trailing: 37×37 pt max; expanded height: 160 pt max per HIG)
- Dynamic Island compact: leading 40×40 pt, trailing 40×40 pt (Apple HIG)

### Surface A — Location Priming Screen (IOS-09, D-01)

**What it is:** A new Flutter screen inserted into the onboarding flow BEFORE the iOS
"When In Use" system prompt. Shown iOS-only (Platform.isIOS gate). Android skips it entirely.

**Primary visual focal point:** The 22px/w700 heading `"Your location stays on your device"` —
the largest, boldest element on screen — anchors the trust message immediately below the icon.

**Layout:** Reuse the same scaffold pattern as `onboarding_screen.dart`:
- `Scaffold` → `SafeArea` → `LayoutBuilder` → `SingleChildScrollView` → sticky-footer column
- Horizontal padding: 32px, top padding: 60px, bottom padding: 32px
- No AppBar

**Content structure (top to bottom):**
1. Location icon — `Icons.location_on_rounded`, size 48px, color `accent`, centered
2. `SizedBox(height: 24)`
3. Heading text: `"Your location stays on your device"`, 22px / w700 / Inter / letterSpacing -0.6
4. `SizedBox(height: 8)`
5. Body copy (max width 280px): `"Traevy records your route to measure traffic time. All trip data is stored on your iPhone — never shared without your consent."`, 16px / w400 / Inter / height 1.5 / textDim color
6. `SizedBox(height: 32)`
7. Three FeatureTick rows (reuse `FeatureTick` widget exactly):
   - Tick 1: title `"Route recording"` / subtitle `"Captures your GPS path in the background."`
   - Tick 2: title `"Speed-based traffic"` / subtitle `"We detect stuck time using speed — no other data."`
   - Tick 3: title `"Device-only storage"` / subtitle `"Trips never leave your iPhone unless you sign in."`
   - Gaps between ticks: `SizedBox(height: 16)` (md token)
8. `Spacer()`
9. Primary CTA button — full-width, same shape/height as `GoogleContinueButton` (14px radius, 16px vertical padding, bgElev background, borderStr border):
   - Label: `"Allow location access"` (Inter 16px w700)
   - Icon: `Icons.location_on_outlined`, 20px, placed left of label (same slot as Google G logo in GoogleContinueButton)
   - On tap: proceed to request `Permission.locationWhenInUse` via `TrackingPermissionService`
10. `SizedBox(height: 8)`
11. Skip link — centered, `"Skip for now"`, 14px / w400 / textMuted
    - On tap: navigate forward without requesting location (degraded mode from first launch)
12. `SizedBox(height: 16)`
13. Terms blurb — centered, `"You can change location access in Settings at any time."`, 14px / w400 / textMuted / height 1.5

**States:**
- Default: all content shown as above
- No loading state — the screen does not wait for anything; it is purely informational

**Notes:**
- Do NOT add a notification priming step here (D-06 / D-07: notification permission is late/contextual)
- The screen is a new `onboarding_location_priming_screen.dart` file; do NOT modify `onboarding_screen.dart` body — insert this screen into the route sequence before the system prompt fires
- After tapping "Allow location access", the iOS system "When In Use" dialog appears immediately

**Accessibility:**
- CTA button: `Semantics(label: 'Allow location access, required to record commutes')`
- Icon at top: `ExcludeSemantics` wrapper (decorative)
- FeatureTicks inherit existing `FeatureTick` accessibility (title is readable, subtitle is hint)

### Surface C — iOS Live Activity: Lock Screen (IOS-13, D-08, D-10)

**What it is:** A native SwiftUI Widget Extension surface, NOT a Flutter widget.
This section specifies the visual contract the native Widget Extension must implement.
The Dart side drives it via a Flutter bridge (live_activities plugin or custom platform channel).

**Layout — Lock Screen expanded presentation (the primary surface):**

```
┌──────────────────────────────────────────────────────────────┐
│  [Direction badge]                    [Moving/Stuck chip]    │
│  TO OFFICE                            ● MOVING               │
│                                                              │
│  38:22                                2.4 km                 │
│  elapsed                              distance               │
│                                                              │
│  ┌──────────────────────────────────┐                        │
│  │          Stop commute            │  ← destructive button  │
│  └──────────────────────────────────┘                        │
└──────────────────────────────────────────────────────────────┘
```

**Field specifications:**

| Field | Value source | Format | Typography |
|-------|-------------|--------|------------|
| Elapsed time | `TripAccumulator.elapsedSeconds` | `HH:MM:SS` if ≥ 1hr, else `MM:SS` — JetBrains Mono equivalent (SwiftUI `.monospaced`) | 28pt semibold |
| Elapsed label | static | `"elapsed"` | 11pt regular, .secondary |
| Distance | `TripAccumulator.distanceMeters / 1000` | `"X.X km"` (1 decimal, always km) | 22pt semibold mono |
| Distance label | static | `"distance"` | 11pt regular, .secondary |
| Direction badge | `TripAccumulator.direction` | `"TO OFFICE"` or `"TO HOME"` — uppercase | 11pt semibold, caps badge, accent-bg fill, accent-foreground |
| Moving/stuck chip | `TripAccumulator.isMoving` (speed ≥ kStuckSpeedThresholdKmh) | `"● MOVING"` or `"● STUCK"` — uppercase | 11pt semibold; moving color if MOVING, stuck color if STUCK |
| Stop button | App Intent | `"Stop commute"` | 13pt semibold, system red fill, white label, full-width, 12pt corner radius |

**States:**

| State | Description |
|-------|-------------|
| Live (default) | All fields populated from snapshot stream; elapsed ticks each second |
| Stale (stream gap) | Elapsed continues ticking via ActivityKit timer; other fields freeze at last value |
| Ending | `isStale: true` passed in ActivityKit update → system shows "Ended" label overlay at dismissal |

**Update cadence:** ActivityKit `update(using:)` called on every `TripAccumulator` snapshot (1 Hz via `kTrackingUiUpdateInterval`). ActivityKit itself may throttle in the background; elapsed should use `Text(timerInterval: ...)` / a `Date` anchor so it ticks client-side without a push.

**Dismissal:** Call `Activity.end(dismissalPolicy: .immediate)` when `TrackingServiceController.stop()` completes.

**Spacing inside the expanded lock-screen view (SwiftUI):**
- Outer padding: 16pt all sides
- Elapsed / distance section: HStack with Spacer between, top half of view
- Direction badge and moving/stuck chip: HStack, bottom edge of top section
- Stop button: bottom of view, height 44pt minimum (HIG touch target)

### Surface E — Android "Active Commute" Notification Enrichment (IOS-14, D-12)

**What it is:** Additive enrichment of `TrackingNotificationService.showRecording()`. The existing

**Notification appearance — unchanged:**
- Channel: `kTrackingNotificationChannelId` (no new channel)
- Notification ID: `kTrackingNotificationId` (same; no duplication risk)
- `ongoing: true`, `autoCancel: false`, `onlyAlertOnce: true` — all preserved

**CONSTRAINT:** `Platform.isAndroid` gate in `showRecording()` — do NOT call `showRecording()`
when `Platform.isIOS` (D-11). The gate belongs in `TrackingServiceController` or
`TrackingNotificationService.showRecording()` itself.

**Parity visual result:** Both the iOS Live Activity and the Android enriched notification
now show the same four data points: elapsed time, distance, moving time, stuck time.

## Data Field Formatting

Live Activity and Android notification both render the same four data points. Formatters
must produce identical output on both surfaces.

| Field | Source | Format rule | Example |
|-------|--------|-------------|---------|
| Elapsed | `elapsedSeconds: int` | `MM:SS` below 1hr; `H:MM:SS` at/above 1hr | `"38:22"`, `"1:04:15"` |
| Distance | `distanceMeters: double` | `(m / 1000).toStringAsFixed(1) + ' km'` | `"2.4 km"` |
| Moving time | `timeMovingSeconds: int` | `_formatStuck(seconds)` — existing helper: `Xm` under 1hr, `XhYm` above | `"34m"`, `"1h02m"` |
| Stuck time | `timeStuckSeconds: int` | Same `_formatStuck(seconds)` helper | `"4m"` |
| Speed (moving/stuck decision) | `currentSpeedMs` compared to `kStuckSpeedThresholdMs` | Boolean: `currentSpeedMs >= kStuckSpeedThresholdMs` → MOVING | — |

The `_formatStuck` helper already exists in `TrackingNotificationService`. Extract it to
`lib/shared/utils/formatters.dart` (or reference the existing one) so the Android notification
and any Dart-side bridge code share the same implementation.

The SwiftUI Live Activity receives pre-formatted strings from the Flutter bridge
(`ActivityAttributes.ContentState`) so it does not need to re-implement formatting logic.
Bridge pushes: `elapsedFormatted: String`, `distanceFormatted: String`,
`movingFormatted: String`, `stuckFormatted: String`, `isMoving: Bool`, `direction: String`.

## Interaction Flow: iOS Permission Sequence

```
OnboardingScreen (existing — auth)
    ↓  after sign-in / skip
LocationPrimingScreen (NEW — Surface A, iOS only)
    ↓  tap "Allow location access"
iOS system "When In Use" dialog (native — not designed here)
    ↓  granted
MainShell / Dashboard (existing)

    ↓  user taps Start for first trip
iOS system "Always Allow" dialog (native — not designed here)
    ↓  always granted → no banner
    ↓  when in use only → Surface B banner on tracking screen
```

**iOS < 17 tracking flow:**
- Trip Start: blue location indicator in status bar (iOS system UI — not designed here)
- No Live Activity is created
- Stop remains in-app (existing tracking screen Stop button — unchanged)

**iOS 17+ tracking flow:**
- Trip Start: Live Activity created (Surface C/D appears on lock screen and in Dynamic Island)
- Activity updates at 1 Hz via bridge
- Trip Stop (in-app or via Live Activity Stop App Intent): Activity dismissed, trip saved

## Native iOS Extension Contract

**Extension target name:** `TraevyLiveActivity` (suggested; consistent with bundle ID pattern `com.travey.app.TraevyLiveActivity`)

**Files required:**
- `ios/TraevyLiveActivity/TraevyLiveActivityWidget.swift` — WidgetKit / ActivityKit widget
- `ios/TraevyLiveActivity/TraevyLiveActivityAttributes.swift` — `ActivityAttributes` conforming struct
- `ios/TraevyLiveActivity/Localizable.strings` — copywriting strings (see above)
- `ios/Runner/Info.plist` — add `NSSupportsLiveActivities: YES` and `NSSupportsLiveActivitiesFrequentUpdates: YES`

**ActivityAttributes contract (Dart → Swift bridge shape):**

```swift
struct TraevyLiveActivityAttributes: ActivityAttributes {
    // Static (set once at start, does not change)
    struct ContentState: Codable, Hashable {
        var elapsedFormatted: String   // e.g. "38:22"
        var distanceFormatted: String  // e.g. "2.4 km"
        var movingFormatted: String    // e.g. "34m"
        var stuckFormatted: String     // e.g. "4m"
        var isMoving: Bool
        var direction: String          // "to_office" | "to_home"
        var startDate: Date            // for client-side elapsed Text(timerInterval:)
    }
    // No static attributes needed — all state is dynamic for this use case
}
```

**Bridge call on each TripAccumulator snapshot:**
1. Flutter side (Dart) formats the four values using existing formatters
2. Calls bridge method `updateLiveActivity(ContentState)` — encodes as JSON
3. Swift side decodes and calls `Activity<TraevyLiveActivityAttributes>.activities.first?.update(using: newState)`

**Stop App Intent:**
- `struct StopCommuteIntent: AppIntent` — calls back into Flutter via `FlutterMethodChannel` or plugin
- Method channel message: `"stopTracking"` → routes to `TrackingServiceController.stop()`
- Intent display name: `"Stop commute"` (matches button label)

**iOS version gate (Dart side):**
```dart
if (Platform.isIOS) {
  final version = int.tryParse(
    (await DeviceInfoPlugin().iosInfo).systemVersion.split('.').first
  ) ?? 0;
  if (version >= 17) {
    // start Live Activity
  }
  // else: blue indicator only — no action
}
```

## Assumptions

The following decisions were made by inference (marked "assumption") rather than explicitly
answered in CONTEXT.md. They are flagged for review during planning.

1. **Assumption: Live Activity elapsed uses `Text(timerInterval:)` for client-side ticking.**
   The ActivityKit bridge updates at 1 Hz, but iOS may throttle background updates.
   Using SwiftUI `Text(timerInterval: startDate...Date.distantFuture)` ensures the elapsed
   clock ticks smoothly even when bridge updates are throttled. The bridge still sends
   absolute `startDate` on Activity creation. This is the Apple-recommended pattern for
   live timers in WidgetKit.

2. **Assumption: Stop App Intent is direct (ends trip, no deep-link).**
   CONTEXT.md Claude's Discretion: "Whether the Stop App Intent ends the trip directly
   or deep-links + ends." This spec chooses direct — the App Intent invokes a
   `FlutterMethodChannel` call that routes to `TrackingServiceController.stop()`.
   No app launch or deep-link needed. If the app happens to be foregrounded, the normal
   stop path fires; if backgrounded, the method channel call wakes the Flutter engine.

3. **Assumption: "1 week of usage" anchor = 7 days since first recorded trip, not first launch.**
   CONTEXT.md D-07: anchor is Claude's discretion. First recorded trip is the better signal
   (a user who installed but never tracked has no use for notification summaries yet).
   If no trip exists after 14 days of install, also trigger on first weekly-summary attempt.

4. **Assumption: Direction badge shows `kDirectionToOffice` / `kDirectionToHome` literals
   rendered as `"TO OFFICE"` / `"TO HOME"` (uppercase, SwiftUI `.textCase(.uppercase)`).**
   The bridge passes the raw direction string and SwiftUI applies casing styling.

5. **Assumption: Android notification enrichment does NOT change the `onlyAlertOnce: true`
   or `Importance.low` settings.** The enriched two-line body is a visual-only change;
   no behavioral regression.

6. **Assumption: The Live Activity `live_activities` pub plugin is evaluated during research
   phase.** This UI-SPEC does not mandate plugin vs. custom platform channel — that is
   an implementation decision for the planner. The contract above (ActivityAttributes struct,
   bridge method names) is compatible with either approach.
